# Agent-Company — IA / Wireframe V1

## 1. 设计目标

第一版不是做“所有功能”，而是先把这三个问题回答清楚：

1. 用户一打开页面，第一眼看到什么？
2. 怎么一眼看懂“这是一家正在运转的 Agent 公司”？
3. 怎么让角色、状态、总结三件事自然地组织在一起？

因此，V1 的核心目标不是复杂，而是：
- 一眼有世界观
- 一眼看见主人与成员
- 一眼看懂公司在运转
- 结构足够稳，后续方便接真实数据

---

## 2. 首页信息架构

首页采用 **1 个主空间 + 3 个辅助层** 的结构。

### 2.1 主空间（最重要）
主空间承担“记忆点”和“产品灵魂”。

这里不是普通 hero，而是：
- 公司主人（当前案例为菲比）
- Agent 办公空间 / 公司舞台
- 正在工作的主要 Agent
- 当前最重要的状态感知

它负责回答：
> 这是谁的公司？现在谁在这里？大家在干什么？

### 2.2 状态流层
状态流层用于展示系统正在发生什么。

内容包含：
- 谁刚上线
- 谁刚签到
- 谁切换了状态
- 谁离线了
- 谁提交了总结
- 谁发生了异常 / 恢复

它负责回答：
> 公司不是静态页面，而是一个正在运转的空间。

### 2.3 成员层
成员层用来展示“公司里都有谁”。

内容包含：
- 成员头像 / 立绘
- 名称
- 身份（主人 / 成员 / 访客）
- 当前状态
- 简介 / 标签

它负责回答：
> 这些 Agent 不是匿名节点，而是有身份的成员。

### 2.4 总结层
总结层用于展示“今天产出了什么”。

内容包含：
- 今日总结
- 最近汇报
- 最新产出
- 成员工作摘要

它负责回答：
> 这家公司不只是在线，而且真的在工作。

---

## 3. 第一版首页布局

采用偏展示型的布局，而不是后台控制台布局。

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部信息条                                                   │
│ 公司名 / 时间 / 在线人数 / 今日概览 / 快速入口               │
├──────────────────────────────────────────────────────────────┤
│ 左侧成员带        │ 主空间舞台                               │ 右侧状态流 │
│ 成员卡 / 标签     │ 菲比主视觉 + 办公空间 + 在线 Agent        │ 最近动态   │
│ 快速切换          │                                           │ 状态变化   │
│                   │                                           │ 异常提醒   │
├──────────────────────────────────────────────────────────────┤
│ 底部总结层                                                   │
│ 今日总结 / 最近汇报 / 最新产出 / 简报入口                    │
└──────────────────────────────────────────────────────────────┘
```

## 4. 各区域功能定义

## 4.1 顶部信息条
目标：给页面一个“公司系统”的骨架感。

建议字段：
- 公司名称：Agent-Company
- 当前主人：菲比（当前案例）
- 当前时间
- 在线人数
- 今日状态概览
- 快速入口（成员 / 总结 / 配置）

注意：
- 顶部信息条不应太重
- 不要做成传统后台导航栏
- 更像“公司抬头 + 状态总览”

## 4.2 左侧成员带
目标：让成员随时可见。

建议内容：
- 成员迷你卡片
- 在线状态点
- 身份标签
- 点击展开查看完整资料

建议优先级：
- 主人永远固定在最上或最突出位置
- 在线成员优先展示
- 访客可弱化显示

## 4.3 中央主空间
目标：形成产品最强记忆点。

中央区域必须承载：
- 公司主人（当前案例为菲比）
- 办公空间
- 主要 Agent 的存在感
- 当前工作氛围

这个区域不建议堆太多控件。

应该更像：
- 被观看的主场景
- 一个轻游戏化舞台
- 一个数字办公室切片

## 4.4 右侧状态流
目标：让页面“动起来”。

建议内容：
- 11:03 菲比切换为工作状态
- 11:01 Claude-Dev 提交今日总结
- 10:58 Gemini-Research 上线
- 10:46 某成员离线

建议设计：
- 时间短句
- 节奏轻快
- 不要写成长日志
- 更像“公司动态流”

## 4.5 底部总结层
目标：把“产出感”放出来。

建议内容：
- 今日总结卡
- 最新汇报卡
- 最近产出摘要
- 一键查看完整日报

底部层不一定永远展开，未来可做成：
- 默认显示 2~3 条摘要
- 点击进入完整总结视图

---

## 5. 页面导航结构

V1 不建议做复杂导航。

建议采用轻导航：
- Home（公司主空间）
- Members（成员）
- Updates（状态流 / 动态）
- Reports（总结）
- Settings（配置）

如果首页做得够强，很多操作甚至可以在首页完成，不必频繁跳页。

---

## 6. 角色层级规则

## 6.1 主人（Owner）
- 页面里最重要的人物
- 当前案例是菲比
- 在主空间中拥有固定主位
- 在成员层中拥有最高优先级展示

## 6.2 成员（Member）
- 正式公司成员
- 可长期存在
- 有稳定资料与身份标签

## 6.3 访客（Guest）
- 临时加入
- 可弱化展示
- 重点体现“来访”而不是“常驻”

---

## 7. 视觉方向（V1）

### 7.1 风格主张
- 像素风
- 温暖公司感
- 不是冷后台
- 不是纯游戏 UI
- 是介于“作品展示”与“数字办公空间”之间的表达

### 7.2 视觉重点
- 菲比主视觉要有温度
- 办公空间要有层次
- 成员卡片要轻，不抢主空间
- 状态流要像“实时动态”而不是审计日志

### 7.3 颜色倾向
建议从菲比的角色气质出发：
- 金色 / 柔黄：主人与主视觉强调
- 浅蓝 / 晴空色：辅助信息与空间气氛
- 暖灰 / 墨蓝：背景与结构骨架

---

## 8. 第一版落地建议

V1 最先做的不是全部页面，而是：

1. 首页静态原型
2. 主空间布局
3. 成员迷你卡片
4. 状态流组件
5. 总结摘要组件

也就是说，先把 **首页打穿**，再扩展子页面。

---

## 9. 结论

Agent-Company 的 V1 应该从首页开始，把它做成一个：

> 一打开就能感受到“菲比正在管理一家公司，而这家公司里的 Agent 正在工作”的页面。

这比先做后台表格、更适合这个产品的灵魂。
