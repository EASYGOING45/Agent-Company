# Agent-Company — Wireframe Notes V1

## 1. 目标

把首页从“模块级结构”进一步推进到“可以开始画静态原型”的程度。

这份文档不做高保真设计，而是定义：
- 每个区域大概占多大
- 信息重心怎么分布
- 首页第一屏的阅读路径是什么

---

## 2. 首页第一屏阅读路径

建议用户第一眼阅读顺序：

1. **公司身份**
   - 这是 Agent-Company
   - 当前主人是谁（当前案例为菲比）
2. **主空间舞台**
   - 看到菲比与办公空间
   - 感受到“公司在运转”
3. **当前活跃成员**
   - 左侧成员带快速感知谁在线
4. **最近发生的事**
   - 右侧状态流感知动态
5. **今天做了什么**
   - 底部总结层承接产出

---

## 3. 首页宽屏布局建议

## 3.1 版式占比
建议采用：
- 左侧成员带：18%~22%
- 中央主空间：52%~58%
- 右侧状态流：20%~24%
- 底部总结层：整宽，约占首屏下方 18%~24% 高度

这样可以保证：
- 中央舞台始终是主角
- 左右两侧辅助信息不压住舞台

## 3.2 顶部条高度
建议较薄：
- 56~72px 左右的系统条感觉
- 不做厚重导航

---

## 4. 区块细化

## 4.1 Top Bar
包含：
- 左：公司名 / 当前主人
- 中：时间 / 在线人数 / 今日概览
- 右：轻入口（成员 / 总结 / 设置）

建议：
- 左重右轻
- 中部概览不要太像 KPI 仪表盘

## 4.2 Member Rail
建议内部结构：
- 顶部：成员区标题
- 中部：成员迷你卡列表
- 底部：查看全部成员入口

每个成员卡建议字段：
- 头像
- 名称
- 身份标签
- 状态点
- 短状态文案（可选）

## 4.3 Main Stage
建议内部结构：
- 背景空间
- 主人区域（菲比当前案例）
- 2~4 个活跃 Agent 的空间点位
- 少量氛围状态元素

不建议：
- 直接堆十几个 Agent
- 把状态流文字塞进舞台里
- 把大量卡片压在舞台上方

## 4.4 Activity Feed
建议内部结构：
- 标题：最近动态
- 时间序列短句列表
- 重要事件稍微高亮
- 底部可有“查看更多”

## 4.5 Summary Dock
建议内部结构：
- 左：今日总结摘要
- 中：最新汇报
- 右：最近产出 / 快捷入口

---

## 5. 手机端思路（先定原则）

手机端不建议简单缩放桌面版。

建议顺序：
1. Top Bar
2. Main Stage
3. Activity Feed
4. Member Rail
5. Summary Dock

也就是：
- 先看主空间
- 再看动态
- 再看成员与总结

---

## 6. 当前结论

首页 wireframe 的核心不是“把信息塞满”，而是：

> 让用户一进来先感受到菲比和她的公司，再自然读到成员、动态和产出。
